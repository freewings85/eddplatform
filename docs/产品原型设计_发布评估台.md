# 产品原型设计：发布评估台（ReleaseLab）

> 版本 v0.1（低保真线框）｜ 日期 2026-07-02
> 依据：`版本化沙箱发布评估平台_基于开源的二次开发架构.md`
> 目的：把"版本化沙箱 + 用例驱动发布评估"落成一个**产品**的样子，先对齐交互思路，再做高保真。

---

## 1. 产品一句话
**用同一批用例，把整个系统的"老版本 vs 新版本"跑出一张标准化对比结果——谁在哪些用例上更好、更差、退化，不凭感觉。**

## 2. 目标用户（角色）
- **业务/PO**：看对比结论、定成功标准、拍板放量。
- **Agent 工程师**：建系统版本、发起评估、下钻 trace 定位问题。
- **Eval/质量**：维护用例库与评估器、校准 judge。
- **平台**：管环境、权限、配额。

## 3. 设计思路（5 条原则）
1. **一个入口**：所有动作在这一个门户完成（承载于 Backstage/独立薄壳）；Langfuse/Grafana 只作为**下钻里的 embed**出现，不让人跳网址。
2. **"对比"是产品的心脏**：整个产品围绕"同一用例集 · 老新并排 · 逐条 improved/regressed"组织。
3. **可追溯下钻**：聚合 → 逐用例 → 输出 diff → trace diff → judge 判分说明，一路点进去。
4. **版本 = 服务镜像 tag 组合**：把"系统版本(v1/v2)"做成一等对象。
5. **环境生命周期可见且短命**：建/跑/销的沙箱状态、TTL 可见，跑完自动销毁。
6. **薄壳复用开源**：界面里的用例/评分/trace 来自 Langfuse、环境来自 Garden/k8s、版本来自 Harbor——门户只是"薄壳 + 编排入口"，不自研能力。

## 4. 信息架构（主导航）
`概览` · `用例库` · `系统版本` · `发布评估` · `对比` · `环境` · `评估器` · `设置`

## 5. 核心用户流程
> 建/选系统版本(v2) → 选用例集 → 选隔离级别 → 一键发起 → 平台自动[拉沙箱 v1&v2 → 部署 → 跑用例 → 评分 → 对比 → 销毁] → 看对比结论 → 下钻回归项 → 决策。

## 7. 与开源/架构的映射（薄壳原则）
| 界面元素 | 背后开源 | 门户只做 |
|---|---|---|
| 用例库 / 评分 / judge / trace 下钻 | **Langfuse** | 调 API + embed |
| 系统版本 / 镜像 tag | **Harbor** | 拉 tag 列表、渲染 manifest |
| 建/销沙箱环境 | **Garden + k8s**(namespace/vCluster) | 触发 CLI/API |
| 流水线进度 | **Temporal** | 展示 workflow 状态 |
| 隔离级别 | **NetworkPolicy / vCluster / Kata** | 传参数 |
| 门户外壳 / SSO | **Backstage** | 自定义 Scaffolder action |
