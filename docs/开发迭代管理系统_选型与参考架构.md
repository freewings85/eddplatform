# AI 开发迭代管理系统：选型、Build-vs-Buy 与参考架构

> 版本 v1.0 ｜ 日期 2026-07-02 ｜ 负责人：AI 团队开发负责人
> **本文回答的问题**：用什么"系统"来管理团队 AI 研发的全流程迭代（需求→设计→开发→测试→评估→迭代），**评估+迭代闭环为核心**；开源拼装还是自研？
> 证据来源：第四轮 deep research（联网检索 + 对抗式交叉验证）。关联：`AI业务全流程研发体系_SOP.md`（本系统是这套 SOP 的"运行载体"）。

---

## 核心判断（先说结论）

**1. 没有单一开源系统能覆盖全生命周期。** 经验证，市面上没有任何一个开源系统把"需求↔设计↔评估↔trace↔迭代"串成一体。**结论：分层拼装 best-of-breed 开源 + 一层薄薄的自研胶水，不要自研大一统平台。**

**2. 胶水是现成的标准，不用自己发明——OpenTelemetry。** pydantic-ai 内置 OTel 埋点，通过 OTel 的 **GenAI 语义约定（`gen_ai.*`）** 把 trace 以厂商中立格式导出，任何 OTLP 后端都能接。**这意味着埋点一次、评估/可观测后端可随意替换，不锁定**——这正是整套拼装能低成本落地的关键。

**3. 推荐的"评估+迭代枢纽"是 Langfuse（可自托管）。** 它原生吃 pydantic-ai 的 OTel span（无需适配代码），且自带迭代所需的全部原语：**prompt 版本管理、eval 数据集、LLM-as-judge、评分、Playground**。Apache-2.0 的备选是 **MLflow 3.x**。

**4. 自研只做"一层薄胶水"**：把"工单/需求 → eval 数据集 → 回归运行 → 灰度放量"用 webhook/OTLP 串起来。其余全用开源。

---

## Build-vs-Buy 判断

| 维度 | 结论 |
|---|---|
| 有没有开源大一统？ | **没有**（本轮无任何"全生命周期一体化开源系统"通过验证）|
| 自研大一统？ | **不建议**——重复造评估/可观测/ALM/灰度这些成熟轮子，成本高、维护重 |
| 推荐 | **分层拼开源 + 自研薄胶水层**（胶水靠 OTel/REST/webhook，工作量小）|

> ⚠️ 说明：本判断与参考架构是对"各层已验证能力事实"的**综合推断**，不是对某套端到端部署的独立基准测试。

---

## 分层选型（每层给：推荐开源 / 能力 / License·自托管 / 与 pydantic-ai 集成 / 注意点）

### 第 0 层 · 埋点（Instrumentation）
- **pydantic-ai + Logfire SDK → OTel `gen_ai.*` span**
- **不锁定**：Logfire 基于 OTel 标准，用 `OTEL_EXPORTER_OTLP_ENDPOINT`（及 TRACES/METRICS/LOGS 变量）可导到任意 OTLP 后端；`send_to_logfire=False` 则只发第三方后端。
- 注意：传输是 HTTP/Protobuf（端口 4318，非 gRPC 4317）；默认会双发，直到设 `send_to_logfire=False`；埋点本身仍用 Logfire SDK。

### 第 1 层 · 评估 + 迭代枢纽（★ 系统核心）

**首选：Langfuse（自托管）**
- **能力**：prompt/版本管理、eval 数据集创建、LLM-as-a-judge 评估器、评分(scores)、Playground——**迭代闭环原语齐全**。
- **集成**：原生吃 pydantic-ai OTel span，`instrument=True` + `get_client()`，**无需适配代码**。
- **License/自托管**：仓库 MIT（`/ee` 除外），可完整自托管。
- ⚠️ 注意：自托管 OTLP 接入需 **Langfuse v3.22+/v3**（v2 没有）；部分企业模块（SCIM、审计日志、数据保留策略）**即使自托管也需商业 license key**；少量用户反馈 pydantic-ai 的 trace 有 input/output 为空的保真度问题——**试点时要实测**。

**备选：MLflow 3.x（Apache-2.0）**
- 既能以 `gen_ai.*` 格式导出，也能通过 OTLP `/v1/traces` 端点**接入** OTel 标准 trace 并按 GenAI 语义一等呈现。
- ⚠️ 版本门槛：`gen_ai.*` 导出需 **3.11+**（设 `MLFLOW_ENABLE_OTEL_GENAI_SEMCONV`），OTLP 接入需 **3.6.0+**；需 SQL 后端；仅 OTLP/HTTP（暂无 gRPC）；需 `x-mlflow-experiment-id` 头。
- 何时选它：更看重 Apache-2.0（无 open-core 顾虑）、或已有 MLflow/ML 实验体系。

> ⚠️ 关于"Langfuse vs MLflow vs Phoenix 谁功能更全/谁没有 open-core 门槛"——本轮此类**厂商横评宣传均未通过验证（被证伪）**，不要据此下结论；以官方文档 + 自己 POC 为准。

### 第 2 层 · ALM / 需求-任务-发布

**首选：Plane（自托管）**
- **MIT** 许可；官方 **MCP server**（100+ 工具、~20 模块：工作项 CRUD、周期/迭代、模块、里程碑、工时…）+ **REST API + OAuth（含 "Agent Run" scope）+ 事件 webhook（HMAC）**。
- → 把工单/需求用 API/webhook 接到 eval/实验系统很顺手。

**备选：OpenProject（自托管）**
- **REST APIv3**（个人 API token、可配 CORS）+ **出站 webhook**（项目、工作包、评论、工时、附件）。
- ⚠️ CORS 默认关闭且不允许 `*` 通配（认证资源）。

**不推荐纳入开源核心：GitLab requirements**
- 确有真正的"需求↔测试"可追溯（`requirements.json` 标 passed/failed），但**仅 Ultimate 付费档**，且是 legacy、正迁移到 work items——**作为开源 go-forward 选择偏弱**。

### 第 3 层 · 发布/灰度/Feature Flag（agent/prompt 版本的灰度与 A/B）

- **GrowthBook**：**MIT** open-core，Docker Compose 自托管；feature flag + **统计实验引擎**（CUPED、Sequential、Bayesian、Bandits、SRM）+ 内置产品分析。
- **Unleash**：**AGPL-3.0**，可本地/任意云；flag + **金丝雀/渐进放量** + A/B + kill switch + 按用户/分段定向。
- 两者均 open-core（部分企业功能需 license key），但核心自托管免费。

### 第 4 层 · 胶水（★ 唯一自研部分，薄）
一个小的 **webhook/OTLP 集成 shim**，按 OTel 语义约定为 key，把：
> 工单里的一条需求 → 绑定 eval 数据集 → 触发回归运行 → eval 结果门控灰度放量
串成可追溯闭环。**这是唯一需要自研的部分，工作量小。**

## 推荐参考架构

```
┌──────────────────────────────────────────────────────────────┐
│  ALM 层：Plane / OpenProject   （需求·任务·发布，REST+webhook+MCP）│
└───────────────┬──────────────────────────────────────────────┘
                │ webhook/REST（需求↔eval 绑定）
┌───────────────▼───────────  自研薄胶水层（OTLP/webhook shim）───┐
│  需求 → eval 数据集 → 回归运行 → eval 门控 → 灰度放量  可追溯     │
└───┬───────────────────────┬───────────────────────┬───────────┘
    │ OTel gen_ai.* span     │ eval API              │ flag API
┌───▼─────────────┐  ┌───────▼──────────┐   ┌────────▼──────────┐
│ 埋点             │  │ 评估+迭代枢纽(核心)│   │ 灰度/AB            │
│ pydantic-ai      │  │ Langfuse          │   │ GrowthBook/Unleash│
│ + Logfire(OTel)  │→ │ (prompt版本/数据集 │   │ (prompt/agent版本  │
│ send_to_logfire  │  │ /LLM-judge/评分/  │   │  灰度门控)         │
│ =False 可改后端  │  │ Playground)       │   │                   │
└──────────────────┘  │ 备选 MLflow 3.x   │   └───────────────────┘
                      └───────────────────┘
   贯穿：OpenTelemetry GenAI 语义约定(gen_ai.*) —— 厂商中立、后端可换、不锁定
```

**数据流（一次迭代）**：埋点产生 `gen_ai.*` trace → Langfuse 收集并可视化 → 从真实 trace 挑失败样本入 eval 数据集 → 改 prompt/工具（Langfuse 管版本）→ 跑 eval 回归 → 胶水层判定不退化 → GrowthBook/Unleash 灰度放量 → 新 trace 回流……工单侧（Plane）全程关联需求，形成**可追溯闭环**。

## 生命周期各阶段 → 系统模块映射（呼应 SOP）

| SOP 阶段 | 由哪个系统承载 |
|---|---|
| 需求管理 | Plane/OpenProject（需求·成功标准·backlog）|
| 详细设计 | Plane 文档/工作项 + 复用登记（可挂 Langfuse dataset 链接）|
| 开发 | Git/GitLab（代码/PR）+ Langfuse（prompt 版本）|
| 测试 | Langfuse 数据集 + 评估器（+ 单点评测库如 promptfoo/DeepEval 可选）|
| **评估** | **Langfuse**（eval 运行、回归对比、评分、LLM-judge）★ |
| **迭代** | **Langfuse**（trace 错误分析→回灌）+ GrowthBook/Unleash（灰度）★ |
| 度量看板 | Langfuse scores + 自建 DX Core 4/DORA 看板（数据可从各系统 API 汇聚）|

## 拼开源 vs 自研：成本结论
- **拼开源**：每层都有成熟自托管方案，且都讲 OTel/REST/webhook → 集成成本低；自研仅"薄胶水"。**推荐。**
- **自研大一统**：要重做评估引擎、trace 存储/可视化、prompt 版本、灰度引擎、ALM——**重复造轮子，性价比差**。
- **净判断**：**采用开源分层 + 自研薄胶水层**。把工程投入省下来的部分，投到"业务差异化 + 复用底座 + 胶水打通"上。

---

## 落地路线（MVP → 完整）
1. **MVP（先跑通评估+迭代核心）**：自托管 **Langfuse**（≥v3.22）+ pydantic-ai `instrument=True` 接入 → 立刻拥有 trace + prompt 版本 + eval 数据集 + LLM-judge。
2. **加灰度**：接 **GrowthBook 或 Unleash**，让 prompt/agent 版本可灰度、可回滚、可 A/B。
3. **加 ALM 打通**：**Plane**（或 OpenProject）+ 写**薄胶水**（webhook：工单需求 ↔ eval 数据集/回归；eval 门控 ↔ flag 放量）。
4. **加度量看板**：从各系统 API 汇聚成 DX Core 4/DORA 看板。
5. **固化**：把这套接线写进 golden path 脚手架（见 SOP §7.5），新业务线开箱即用。

## 待评估候选（本轮未验证，非"不合适"）与残留缺口
- **评估平台候选（需再评一轮再排名）**：Arize Phoenix、Comet Opik、Agenta、Helicone、promptfoo、DeepEval、Ragas、Argilla、Literal AI——本轮无验证结论，**不代表不好**，只是没在本轮排名。若要在 Langfuse/MLflow 之外再横评，建议专门补一轮 POC。
- **ALM 候选**：Taiga、Redmine、Leantime、Huly（未验证）。
- **灰度/IDP**：Flagsmith（未验证）；**Backstage**（IDP/golden path/软件目录）在"批量造 agent 的标准化"里的具体作用**本轮无验证结论**，值得单独评估（尤其它的 scaffolder 模板可承载 SOP §7.5 的脚手架）。
- **残留缺口**：① OTel GenAI 语义约定 2026 年仍处 experimental（属性名可能变，用 `OTEL_SEMCONV_STABILITY_OPT_IN`）；② eval 回归结果**自动门控** flag 放量的成熟连接器是否存在，需 POC；③ 胶水层真实工程量/维护负担待评估。
