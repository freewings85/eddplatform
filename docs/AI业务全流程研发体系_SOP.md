> 版本 v1.0 ｜ 日期 2026-06-30 ｜ 负责人：AI 团队开发负责人
> **性质**：团队可直接执行的操作手册（操作层）。
> **依据层**：`AI研发提效_调研与落地方案.md`（战略/选型/build-vs-buy/度量的论证与引用都在那里）。
> **技术基线**：Agent 框架 = pydantic-ai；Eval = Pydantic Evals；可观测 = OpenTelemetry → Logfire/Langfuse；持久化执行 = Temporal/DBOS（按需）；AI 编码助手 = Claude Code/Cursor。

## 0. 总览：怎么用这份 SOP

### 0.1 基本单元与主线
- **单元**：一条业务 Agent（如"保险报价咨询""智能客服""门店查询"）。
- **主线**：每条业务 Agent 都走同一条**六阶段流水线**；多业务线靠**共享底座复用**摊薄成本——业务线越多，复用收益越大。
- **核心理念**：**先固化流程与复用，再叠加 AI；AI 在每个阶段是"加速器"，不是"免检通道"。** 提速必须配质量门，否则只是把问题更快推到线上（DORA 2025：AI 提速会暴露下游稳定性问题）。

### 0.2 六阶段流水线（带闭环）

```
① 需求管理 ─→ ② 详细设计 ─→ ③ 开发 ─→ ④ 测试 ─→ ⑤ 评估 ─→ 上线/灰度
     ↑                                                              │
     └──────────────── ⑥ 迭代（trace 错误分析 → 回灌 eval → 复用沉淀）←┘
```

### 0.3 每个阶段统一用 8 段式写（保证一致、可执行）
> 【目标】→【标准步骤(SOP)】→【产出物 + 模板】→【AI 工具怎么提效】→【规范/Checklist】→【质量门】→【角色与 HITL】→【度量指标】

### 0.4 角色矩阵（RACI 简版）

| 角色 | 主要职责 |
|---|---|
| **业务负责人(PO)** | 提业务意图、定优先级、确认成功标准、放量决策 |
| **Agent 工程师** | 设计、开发、prompt/工具/技能实现、自测 |
| **Eval/质量** | eval 数据集、评测器、judge 校准、质量门把关 |
| **平台/基础设施** | 脚手架、共享库、编排/持久化、可观测、CI |
| **安全治理** | 高危工具、权限审批审计、注入/泄露防护、输出审查 |
| **Reviewer** | code review（含 agent 代码冗余/技术债/安全）|

### 0.5 五个横切层（贯穿所有阶段，详见第 7 节）
复用机制 ｜ 度量体系(DX Core 4) ｜ 安全治理 ｜ AI 工具嵌入规范 ｜ Golden Path（标准脚手架）

### 0.6 全流程质量门一览（不达标不进下一阶段）

| 阶段 | 退出质量门（Exit Gate）|
|---|---|
| ① 需求 | 需求卡评审通过；每个意图有**可被 eval 检验**的成功标准；复用预评估完成 |
| ② 设计 | 设计评审通过；工具清单复用率达标；安全设计 review 通过 |
| ③ 开发 | CI 通过（基础 eval+类型+lint）；code review 通过；无降门槛 |
| ④ 测试 | eval 数据集达最小规模（≥~100 条且做过错误分析）；安全测试通过 |
| ⑤ 评估 | eval 回归**不退化** + 安全门 + 灰度稳定性达标 |
| ⑥ 迭代 | 每轮迭代 eval 不退化；失败已回灌；可复用项已沉淀 |

## 阶段 ① 需求管理

**【目标】** 把模糊的业务诉求，变成**可设计、可验证**的 Agent 需求，并锁定**可被 eval 检验的成功标准**。

**【标准步骤】**
1. **意图采集**：业务方提出场景（如"保险报价咨询"）。
2. **边界定义**：in-scope / out-of-scope；happy path + 异常路径 + 高风险路径。
3. **需求拆解**：意图清单、每个意图所需信息、依赖系统/数据、所需工具能力。
4. **复用预评估**（硬规范）：查共享库——哪些工具/技能/记忆/业务语义已存在、可直接复用。
5. **成功标准定义**（关键）：每个意图"什么算答对"，**必须写成可被 eval 检验的判定**（优先二元 pass/fail）。
6. **风险与 HITL 识别**：哪些情形必须转人工/需审批（高风险、不确定、涉及金额/合同）。
7. **入池排期**：纳入需求池，定优先级。

**【产出物 + 模板】** 《Agent 需求卡》：
```
场景名称：
业务价值/目标：
意图清单：[意图1, 意图2, ...]
每意图成功标准(可测)：意图1 → <二元判定描述>
所需信息/槽位：
依赖系统/数据/接口：
所需工具能力：[需复用: ... ｜ 需新建: ...]
复用预评估结果：（命中共享库哪些项）
风险与 HITL 触发点：
验收 eval 草案：（3-5 条典型 case 的输入→期望）
out-of-scope：
```

**【AI 工具怎么提效】**
- 用 LLM 从**历史对话/工单**挖掘高频意图、边界、典型问法（人审）。
- 用 AI 把口语化需求**草拟成结构化需求卡**（人改定稿）。
- ⚠️ AI 产出一律人审——需求错会放大到全流程。

**【规范/Checklist】**
- [ ] 每个意图的成功标准**能被 eval 检验**（不是"回答得好"这种模糊话）？
- [ ] 是否查过共享库、填了复用预评估？
- [ ] 高风险路径是否定义了 HITL/审批？

**【质量门(退出)】** 需求卡评审通过 + 成功标准可测 + 复用评估完成。

**【角色与 HITL】** 业务负责人(主) + Agent 工程师 + Eval/质量（共同定成功标准）。

**【度量指标】** 需求→设计前置时间；复用预评估覆盖率。

## 阶段 ② 详细设计

**【目标】** 产出可实现的 Agent 设计：架构、工具/技能、prompt/上下文、记忆、HITL、安全、编排。

**【标准步骤】**
1. **架构选择**：workflow（固定流程）vs agent（自主规划）。原则"**能简单别复杂**"——多数业务线先用 workflow，确有自主规划需求再上 agent。
2. **意图/对话流设计**：状态、转移、澄清策略、兜底。
3. **工具 & 技能清单**：列全所需工具；**先查复用库，缺的才新设计**；定义**强类型 I/O**（pydantic 模型——pydantic-ai 天然支持）。
4. **Prompt / 上下文工程设计**：system prompt 结构、上下文装配（"最小高信号 token 集"）、变量与版本策略。
5. **记忆设计**：需要哪些记忆（用户/任务/事实）、读写时机、与 ontology 对齐。
6. **HITL 设计**：转人工触发条件、审批点、人工接管交互。
7. **接口/数据设计**：外部系统调用契约、数据格式。
8. **安全治理设计**：高危工具识别、权限边界、注入防护、输出审查点。
9. **编排/持久化评估**：是否长流程/需断点续跑/长时挂起？→ 是否引入 Temporal/DBOS（durable execution，"应买不应造"，复用开源）。

**【产出物 + 模板】** 《Agent 设计文档(ADD)》：
```
架构选择：workflow / agent（+理由）
意图流图/状态机：
工具&技能清单：| 名称 | 复用/新建 | 输入(pydantic) | 输出(pydantic) | 风险等级 |
Prompt 设计稿：（system 结构 + 上下文装配规则 + 版本策略）
记忆设计：
HITL 点：| 触发条件 | 动作 | 审批人 |
接口/数据契约：
安全设计：| 高危点 | 防护措施 |
编排/持久化：是否需要 + 选型
eval 维度草案：（要测哪些质量维度）
```

**【AI 工具怎么提效】**
- 用 AI 生成**多个架构选项 + 取舍对比**（人定）。
- 用 AI 起草 **prompt 初版 / 工具签名**（人改）。
- 用 AI 做**设计完整性检查**（是否漏了异常/安全/HITL）。

**【规范/Checklist】**
- [ ] 工具清单是否**优先复用**、复用率达标？
- [ ] workflow/agent 选择是否有依据（没滥用 agent）？
- [ ] HITL 与安全是否覆盖所有高风险路径？
- [ ] I/O 是否强类型化？

**【质量门(退出)】** 设计评审通过 + 工具复用率达标 + 安全设计 review 通过。

**【角色与 HITL】** Agent 工程师(主) + 平台(复用/编排评审) + 安全(安全评审)。

**【度量指标】** 设计复用率；设计返工率。

## 阶段 ③ 开发

**【目标】** 按设计实现 Agent，守住质量与复用，安全地用 AI 编码助手加速。

**【标准步骤】**
1. **从脚手架起步**：克隆 **pydantic-ai golden path 脚手架**（标准目录/配置/CI/trace 接线已就位，见 §7.5）。
2. **实现工具/技能**：强类型、原子化、可注册；**复用优先**；新工具完成后**回贡共享库**。
3. **Prompt 实现与版本化**：prompt 进 git，可回滚、可 diff、可对比。
4. **接入记忆/外部系统**。
5. **用 AI 编码助手加速**：配 `CLAUDE.md`/规则文件把团队规范喂给助手；**新人/新代码鼓励用，资深/核心存量代码自主选择**（METR：资深者在熟悉代码库上可能反而变慢）。
6. **自测**：本地跑冒烟 + 基础 eval。
7. **Code Review**：按 agent 代码 review 标准（见 §7.3），**AI 生成代码同等门槛、不降标**。

**【产出物 + 模板】** 可运行 Agent + 工具登记记录 + prompt 版本 + PR。
pydantic-ai 最小骨架（含可观测一行接线）：
```python
from pydantic_ai import Agent
agent = Agent('your-model', system_prompt=..., tools=[...], instrument=True)  # instrument=True 即出 OTel trace
```

**【AI 工具怎么提效】**
- 编码助手生成样板代码、工具实现、测试草稿。
- ⚠️ **度量真实收益，不靠感觉**（METR：自评可与实测相反）——见 §7.4 的 A/B 方法。

**【规范/Checklist】**
- [ ] 是否走脚手架（没绕过 golden path）？
- [ ] 新工具是否登记到共享库、可被别的业务线复用？
- [ ] prompt 是否版本化？
- [ ] review 是否专门查了**冗余/重复**（agent 代码已知弱点）？

**【质量门(退出)】** CI 通过（基础 eval + 类型检查 + lint）+ code review 通过 + 无降门槛放行。

**【角色与 HITL】** 开发者(主) + Reviewer。

**【度量指标】** 复用率（实际复用 vs 新造）；AI 工具采纳率/每周节省工时（A/B）；review 返工率。

## 阶段 ④ 测试

**【目标】** 上线前系统验证功能、Agent 行为、安全；**构建 eval 数据集**（后续评估与回归的基础）。

**【标准步骤】**
1. **传统测试**：单元/集成（工具、接口、确定性逻辑）。
2. **构建 eval 数据集**：来自**真实 trace / 历史工单 / 需求阶段成功标准**；覆盖 happy + 异常 + 高风险。
3. **Agent 行为测试**：多轮对话、工具调用正确性、HITL 是否按设计触发。
4. **评测器设计**：
   - 确定性质量（格式/PII/正则/结构）→ **代码校验器**。
   - 主观质量（准确性/相关性/幻觉）→ **LLM-judge**，且**必须用人工标注校准**（"谁来验证验证者"）。
5. **标注规范**：**优先二元 pass/fail**；评判标准随观察输出而演化（criteria drift，是活的工件）。
6. **安全测试**：提示词注入、越权调用、数据泄露、高危操作。

**【产出物 + 模板】** Pydantic Evals 数据集 + 评测器 + 测试报告。
Pydantic Evals 骨架：
```python
from pydantic_evals import Case, Dataset
dataset = Dataset(
    cases=[Case(name="保险报价-正常", inputs=..., expected_output=..., metadata={"risk":"low"})],
    evaluators=[...],  # 代码校验器 + LLMJudge（已人工校准）
)
report = dataset.evaluate_sync(your_agent_task)   # 结果可入 git / 进 Logfire 看板
```

**【AI 工具怎么提效】**
- 用 AI 从 trace **批量生成候选 eval case**（人审入库）。
- 用 AI 辅助**错误分析归类**（失败模式聚类）。

**【规范/Checklist】**
- [ ] eval case 是否来自**真实数据**（不是凭空编）？
- [ ] 是否覆盖高风险路径？
- [ ] LLM-judge 是否做过**人工标注校准**？
- [ ] 是否用二元标注？

**【质量门(退出)】** eval 数据集达最小规模（**对 ≥~100 条真实 trace 做过错误分析**）+ 安全测试通过。

**【角色与 HITL】** Agent 工程师 + Eval/质量(主) + 安全。

**【度量指标】** eval 覆盖度；judge 与人工标注一致率。

## 阶段 ⑤ 评估

**【目标】** 用统一 eval + 可观测，决定能否上线/灰度/放量。

**【标准步骤】**
1. **eval 套件回归**：每次变更跑全量 eval。
2. **质量门判定**：看**回归是否退化**（核心），而非追 100% 通过率（~70% 才说明 eval 在真正压测；100% 往往是题太简单）。重点盯**新出现的退化项**。
3. **接入可观测**：`instrument=True` → OTel → Logfire/Langfuse，**trace 必采**。
4. **灰度评估**：按用户/比例/白名单上线；线上看 trace + 关键业务指标 + **稳定性**（DORA 提醒：提速会暴露稳定性问题）。
5. **上线 gate**：eval 门 + 安全门 + 灰度观察达标 → 放量。

**【产出物 + 模板】** 评估报告 + 上线/放量决策记录 + trace 看板。
《上线 Gate 检查表》：
```
[ ] eval 全量回归：无退化（列出对比）
[ ] 安全门：注入/越权/泄露测试通过
[ ] 可观测：trace 已接入、看板可见
[ ] 灰度：__% 流量 / __ 天，稳定性指标达标
→ 放量决策：通过 / 驳回（原因）
```

**【AI 工具怎么提效】**
- 用 AI **汇总 eval 失败模式**、生成评估报告初稿。
- 用 AI 辅助 **trace 异常归因**。

**【规范/Checklist】**
- [ ] 回归是否不退化？
- [ ] trace 是否接好、灰度稳定性是否守住？
- [ ] 是否没有为了"通过率好看"而放水？

**【质量门(退出/放量)】** eval 回归不退化 + 安全通过 + 灰度达标。

**【角色与 HITL】** Eval/质量 + 平台 + 业务负责人(放量决策)。

**【度量指标】** eval 通过率趋势；变更失败率/稳定性(DORA)；灰度→全量周期。

## 阶段 ⑥ 迭代

**【目标】** 用线上真实数据持续改进，并把可复用能力**沉淀回底座**（这是多业务线提效的复利来源）。

**【标准步骤】**
1. **线上 trace 错误分析**（定期）：看真实 trace，归类失败模式——**错误分析是 eval 体系里最重要的活动**。
2. **回灌 eval 数据集**：把新失败模式变成新 eval case（防止回归）。
3. **改进**：prompt/工具/流程优化 → 回到阶段 ③–⑤ 小循环。
4. **复用沉淀**：把跨业务可复用的工具/技能/记忆/语义**抽象进共享库**；更新业务 ontology。
5. **度量回看**：DX Core 4 四维 + 结果指标，季度复盘。
6. **judge/标准维护**：定期人工校准 judge，更新评判标准（criteria drift）。

**【产出物 + 模板】** 迭代记录 + 共享库更新 + 度量复盘。

**【AI 工具怎么提效】**
- 用 AI 做 **trace 聚类 / 失败模式发现**。
- 用 AI 起草**复用抽象候选**（哪些工具/技能值得提取到共享库，人定）。

**【规范/Checklist】**
- [ ] 新失败是否**回灌 eval**？
- [ ] 可复用项是否**沉淀**并登记？
- [ ] judge 是否定期校准？

**【质量门】** 每轮迭代 eval 不退化 + 复用沉淀有记录。

**【角色与 HITL】** 全角色；业务负责人主导复盘。

**【度量指标】** 复用率提升；MTTR/稳定性；每业务线迭代周期。

## 7. 横切层（贯穿所有阶段）

### 7.1 复用机制（多业务线提效的核心杠杆）
- **共享库**：工具(Tools) / 技能·工作流(Skills/Workflow) / 记忆(Memory) / 业务语义(Ontology)。
- **硬规范**："**先查后造**"——需求(①)做复用预评估、设计(②)优先复用、开发(③)新建必回贡、迭代(⑥)持续抽象沉淀。
- **度量**：复用率（实际复用项 / 应可复用项）——业务线越多，这个指标越重要。
- 理论锚点：企业级 agentic 开发的瓶颈是"知识架构"而非模型能力（把机构知识做成可复用单元的组织更有优势）——作为视角参考，非定论。

### 7.2 度量体系：映射到 DX Core 4（反虚荣）
| DX Core 4 维度 | 我们的落地指标 |
|---|---|
| 速度 | 新业务线 time-to-launch、需求→上线前置时间 |
| 有效性 | AI 工具每周节省工时、感知交付速率、复用率 |
| 质量 | eval 通过率趋势、缺陷率、**稳定性(DORA)** |
| 业务影响 | 上线业务线数、业务转化/满意度 |

**反虚荣/反滥用护栏（铁律）**：① 绝不用于个人绩效考核；② 速度类指标必须配对冲指标；③ 不对指标设目标/绑奖励；④ 永不用自我报告当提效证据；⑤ 度量服务于投资决策，不是微观管理。

### 7.3 安全治理（贯穿设计/开发/测试/评估）
- 高危工具识别与沙箱；工具调用**权限/审批/审计**。
- 提示词注入、记忆注入检测；越权调用、数据泄露防护。
- 输入检测 → 上下文隔离 → 工具调用校验 → 输出审查（四道）。
- 与 plan.txt 的运维安全治理规划对齐，规模化/平台化阶段重点建设。

### 7.4 AI 工具嵌入规范（编码助手）
- **配置**：`CLAUDE.md`/规则文件把团队规范、复用库位置、review 标准喂给助手；沉淀共享 skills/子代理。
- **分人群策略**：新人/新代码鼓励用；资深/核心存量代码自主（METR 边界）。
- **必做 A/B 度量**：选 1–2 条业务线，分"用/不用 AI 助手"两组，用**客观指标**（任务周期、PR 返工、每周省时）对比，**数周内**出结论再决定是否全面铺开。

### 7.5 Golden Path（标准脚手架——规范落地的关键）
"黄金路径"= 一条铺好的、自助的、默认正确的标准路径，让"做对"成为最省力选择。脚手架建议包含：
```
business-agent-template/
  agent/            # pydantic-ai agent 定义（instrument=True 预置）
  tools/            # 工具（强类型，可注册到共享库）
  prompts/          # 版本化 prompt
  evals/            # Pydantic Evals 数据集 + 评测器
  observability/    # OTel → Logfire/Langfuse 接线
  ci/               # CI：lint + 类型 + 基础 eval 回归
  CLAUDE.md         # 团队规范喂给 AI 编码助手
  README + 需求卡/设计文档模板
```
> 规范靠文档推不动，靠"**走标准路最省事**"才推得动。脚手架就是把这份 SOP 编译成默认行为。
