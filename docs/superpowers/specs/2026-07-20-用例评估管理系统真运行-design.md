# 用例评估管理系统 · 真运行改造（M1-M3）设计

日期：2026-07-20 ｜ 分支基线：`main`@`36b76e5` ｜ 状态：待评审

## 背景与问题

main 线已具备：用例/标签管理（CRUD + 界面）、约定式部署器（git 仓 → k8s，真集群验证过）、
Temporal `RunTaskWorkflow`（前置条件按序拉起系统+评估程序，e2e 脚本验证过）。
但按"评估 chatagent 2.3"的真实意图走查界面，撞到 7 堵墙：

1. 系统列表是 `sample_data.py` 硬编码，无注册入口——真实系统进不了平台；
2. Task 存内存 `_TASKS`，重启即丢；
3. Task 无"执行"动作，Temporal 链路与界面完全没接线；
4. `/api/runs`、`/api/evaluations`、`/api/comparison` 全是静态假数据；
5. 用例导入只吃平台自身导出的 Case JSON，chatagent3 的 evals YAML（group/role/turns/expect）无转换路径；
6. 评估程序（EvalProgram）页只读样例，无注册入口；
7. chatagent3 `2.3-eval` 分支缺约定式部署所需的 `.eddplatform.yaml`/`deploy/chart`/`build.sh`（属被评仓改造，非平台侧）。

## 目标与验收

**总验收**：在 web 界面上：注册系统 → 注册评估程序 → 建数据集/用例 → 建 Task → 点"执行"
→ 平台经 Temporal 拉起 系统+评估程序 → 逐用例跑分 → 界面看到每个用例的结果/分数。
被评对象最终为 chatagent 2.3（各仓 `2.3-eval` 分支），评估逻辑在独立评估程序仓。

**零假数据原则**：交付时平台是**空态**——无预置系统/用例/标签/运行记录；一切数据由用户操作产生。
`sample_data.py` 及所有 seed 调用删除；前端各列表页有明确的空态提示。
（域模型中样例仅存在于测试 fixtures。）

**执行模型（已定，方案 A）**：平台 = Temporal client，评估程序仓 = worker。
平台侧 `RunTaskWorkflow` 立起环境后，逐 case 触发 child workflow——workflow 名与 task queue
均为评估程序注册的 `code`——由评估程序 worker 认领执行并返回结构化 `CaseResult`。

## 里程碑

### M1 · 平台管理 + 执行打通（平台侧地基）

**1. 存储层 MySQL 化（弃 SQLite）**
- 新增 `store/db.py`：PyMySQL 连接工厂；配置环境变量 `EDD_MYSQL_HOST/PORT/USER/PASSWORD/DB`，
  默认 `127.0.0.1:3306 root/root eddplatform`；启动时建库（若无）+ 建表（`CREATE TABLE IF NOT EXISTS`）。
- 表模式沿用现行"**JSON 文档列**"设计（MySQL 5.7 原生 JSON 类型；本机为 5.7.44，utf8mb4）：
  - `cases(system_id, case_id, position, data JSON)`、`tags(system_id, tag_id, ...)`——从 sqlite3 平移；
  - 新增 `systems(system_id PK, data JSON)`、`eval_programs(system_id, program_id, data JSON)`、
    `tasks(system_id, task_id, data JSON)`、`runs(run_id PK, system_id, task_id, status,
    workflow_id, created_at, data JSON)`、`case_results(run_id, case_id, data JSON)`。
- `CaseStore`/`TagStore` 公有接口签名不变，底层换 PyMySQL；删除全部 sqlite 代码路径。
- 测试用独立库 `eddplatform_test`（每测清表）；MySQL 不可达时测试 skip 并说明。

**2. 系统注册**
- `POST/PUT/DELETE /api/systems`（`system_id` 唯一性校验；删除需级联确认：有 task/run 时拒绝）。
- 最小字段：`id`、`name`、`owner`、`description`（modules 等富字段保留模型但不强制）。
- 前端 Systems 页：新建/编辑/删除表单；其余 system 级 API 增加"系统存在"校验（404）。

**3. 评估程序注册**
- `EvalProgram` 补 `code` 字段（RunCase workflow 名 + task queue 的锚点）与 `git_url`/`ref`。
- CRUD API + 前端表单（评估程序页从只读样例改为真实管理）。

**4. Task 持久化 + 执行触发**
- `task_store` 落 MySQL（CRUD 补全：现仅有 list/create，补 update/delete）。
- Tasks 页每行加"执行"按钮 → `POST /api/systems/{sid}/tasks/{tid}/run`：
  1. 校验前置条件完整（start_system/start_eval_program 必须有 git_url+ref；ref 允许运行时覆盖入参）；
  2. 生成 `RunRecord(status=running, workflow_id=edd-run-{run_id})` 落库；
  3. 经 `runtime.temporal.client` **异步 start**（不 await 完成）`RunTaskWorkflow`；
  4. 后台协程 await workflow 结果 → 回写 RunRecord（status/versions/outcomes）。
- 兜底：平台重启丢失后台协程时，`GET /api/runs/{id}` 按 `workflow_id` 向 Temporal describe 补查并回写。
- Temporal 不可达：`/run` 返回 503 + 明确文案（"Temporal server 未启动"），不产生 RunRecord。

**5. Runs 真数据**
- `/api/runs?system_id=`、`/api/runs/{id}` 读库；删除假数据端点数据源。
- Runs 页：列表（状态/任务/版本标签/时间）+ 详情（各前置条件 outcome、namespace、失败 detail）。
- `/api/evaluations`、`/api/comparison` 本里程碑**直接下线**（返回空/410），对应前端页显示"暂未实现"空态
  ——不留假数据；M2 携真实逐用例结果重建。

### M2 · 逐用例评估（方案 A 主体）

- 扩展现有 `RunTaskWorkflow`（不另立新 workflow）：环境就绪后读取 task 关联 dataset 的 cases，
  逐 case `execute_child_workflow(workflow=evalProgram.code, task_queue=evalProgram.code,
  arg=CaseInput{case 数据, 环境入口, namespace, run_id})`，逐条收 `CaseResult` 写 `case_results` 表
  （子 workflow 失败单条记 failed，不整体中断；结束后汇总 run 状态）。
- `CaseInput`/`CaseResult` 契约固化在 `runtime/temporal/shared.py`（评估程序仓按此实现 worker）。
- Runs 详情页：逐用例结果表（case、状态、分数、指标、trace 链接）。
- 用例导入约定：新增"chatagent evals YAML → Case"转换器（`turns/expect` 存入 Case 的结构化字段；
  role/group 映射为标签），导入界面支持该 YAML。
- （重建对比）同一 task 两次 run（不同 ref）→ 逐用例分数对齐的 old-vs-new 对比视图。

### M3 · chatagent 2.3 接入（真实闭环验收）

- chatagent3 `2.3-eval` 分支：补 `.eddplatform.yaml`、`deploy/chart`、`build.sh`（含 mock datamanager、
  `USE_NACOS=FALSE`、OTel/Langfuse 注入等评估期默认值）；orchestrator 仓视需要同法建 `2.3-eval`。
- 新建 chatagent 评估程序仓：Temporal worker（认领 `code` 队列）+ 前门驱动（SSE 聚合）+
  判定（tools/no_tools/judge rubric）+ Langfuse 读真实 token；素材搬自
  `feat/edd-langfuse-true-token` 分支 `examples/chatagent/`。
- 端到端验收：用户在界面完成注册→导用例→执行→看逐用例分数全流程。

## 非目标（本轮不做）

- 2.0（方案 A' 前门 5 进程）接入与跨方案对比——M3 之后；
- Harbor/Backstage/Garden live、镜像仓库、门户 SSO；
- 多用户/权限；数据库迁移工具（表结构直接 CREATE，改表手工处理）。

## 测试与验证策略

- 全程 TDD：存储层打真 MySQL（`eddplatform_test` 库）；API 用 TestClient + 真 MySQL；
  Temporal 用 in-memory `WorkflowEnvironment`（仓内已有先例 `test_temporal_workflow.py` 模式）。
- UI 自验：Playwright 无头驱动真实界面走全流程（空态 → 注册 → 执行 → 看结果）后再交用户验收。
- 交付验收基线：**空界面**，用户从零操作。
