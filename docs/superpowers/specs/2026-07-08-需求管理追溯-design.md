# 设计：需求管理追溯层（Requirement Traceability）

- 日期：2026-07-08
- 状态：设计定稿（原型已更新，待写实现计划）
- 相关：`prototype/index.html`（需求页 / 用例库需求列 / 评估对比「按需求」汇总均已更新）

## 背景与要解决的缺口

现平台的叙事从「已有 SystemVersion / Dataset / Case」开始，相当于从项目生命周期中段切入，缺一根贯穿的追溯线：**为什么做这个版本、某条用例在验证哪个需求、这次评估的改善/回归对应哪个需求的验收**。

补上「需求」这一维，把 `需求 → 用例 → 评估 → 发布决策` 串成可追溯闭环——评估只是这条完整生命周期里的一环。

## 范围与核心原则

**薄追溯层，复用外部工具（Jira），不自研需求管理。** 与平台既有哲学一致：正如评估引擎复用 Langfuse 不自研，需求详情（描述 / 状态 / 流转 / 讨论）留在 **Jira（唯一真相源）**，平台只做 Jira 给不了的那部分——**需求 ↔ 用例 ↔ 评估结果**的连接与达标汇总。

明确**不做**：需求的 CRUD 工作流、状态机、评论。这些在 Jira。

## 关键决策（已与用户逐条确认）

1. **建模方式 = 方案 A「标签内联」**：新增轻量 `Requirement` 锚点；关联用实体上的 `list` 字段（同现有 `evaluator_names` / `applicable_versions` 风格），不建规范化关联表。
2. **追溯范围**：需求同时连 **Case**（验收用例，多对多）和 **SystemVersion**（该版本交付哪些需求）。
3. **平台零状态**：`Requirement` **不带状态字段**。状态以 Jira 为准；想「筛未完成需求」去 Jira 用它的筛选器（本版不在平台内做状态过滤）。
4. **Jira 交互 = 只写 + 关联**：新建需求时可**推送到 Jira 建 issue**（写），回填 Jira 号 & 链接；或**关联已有 Jira 号**；或暂不关联。本版**不从 Jira 拉取**字段。
5. **过滤维度**：用例按需求标签过滤；需求列表按 **Jira 号**过滤。
6. **需求级对比汇总（★本版核心）**：评估对比新增「按需求」视图，把用例级结果卷到需求级。**「达标」定义 = 该需求的验收用例在该版本全部通过**；只统计两版都适用的用例（沿用现有「公平对比」口径）。

### 本版已采用的默认（用户「ok」时未另行指定，均易改）

- 达标判定用**「全部通过」**，未用「通过率阈值（如 ≥80%）」。
- 需求主键用**平台内部号 `R-xxx` + 独立 Jira 号 `PROJ-xxxx`**（未直接用 Jira 号当主键，以便「暂不关联」时也有稳定 id）。
- 需求列表默认展示**候选版本达标**；是否再加「基线达标」列留待实现时按需补。

## 数据模型（`src/eddplatform/domain/models.py`）

新增锚点：

```python
class Requirement(BaseModel):
    id: str                              # 平台内部 id，如 R-101
    system_id: str                       # 系统级（同 Dataset）
    title: str
    description: str | None = None       # 建时填，用于推送到 Jira
    external_key: str | None = None      # Jira 号，如 PROJ-2043
    external_url: str | None = None      # 直接跳 Jira
    note: str | None = None
    # 无 status —— 状态以 Jira 为准，平台不碰
```

在既有模型加标签（多对多，内联 list）：

```python
class Case(BaseModel):
    ...
    requirement_ids: list[str] = []      # 这条用例验收哪些需求

class SystemVersion(BaseModel):
    ...
    requirement_ids: list[str] = []      # 这个版本交付哪些需求
```

需求级汇总结果模型：

```python
class RequirementRollup(BaseModel):
    requirement_id: str
    title: str
    external_key: str | None = None
    total_cases: int          # 两版都适用的验收用例数
    baseline_passed: int
    candidate_passed: int
    # 派生：baseline_met / candidate_met = 是否全通过（total_cases>0 且 passed==total）
    # verdict：达标(未达→达) / 回归(达→未达) / 保持 / 仍未达标

class Comparison(BaseModel):
    ...
    by_requirement: list[RequirementRollup] = []
```

**概念澄清**：「Jira 状态」与「达标」正交。Jira 状态 = 需求在研发流程走到哪；「达标」= 验收用例在某版本评估是否全过，**只有平台知道**、实时算出、不落库、不写回 Jira。Jira 可能标 Done 而评估发现回归 → 未达标，这正是平台价值。

## 评估内核（`src/eddplatform/evals/engine.py`）

新增纯函数，卷积用例级结果到需求级；只算两版共有的用例（与现有 `compare()` 的 `common` 口径一致）：

```python
def rollup_by_requirement(
    baseline: EvalResult, candidate: EvalResult,
    cases: Sequence[Case], requirements: Sequence[Requirement],
) -> list[RequirementRollup]:
    ...
```

`compare()` 可选地填充 `Comparison.by_requirement`（或由调用方组装），保持函数职责单一。

## Jira 适配器（`src/eddplatform/integrations/jira.py`）

薄、只写、可选、离线兜底——**完全照 `evals/adapters/langfuse.py` 的模式**：

```python
def available() -> bool                                   # 看 JIRA_URL/JIRA_EMAIL/JIRA_TOKEN + httpx
def create_issue(project, summary, description) -> dict   # 推送，返回 {key, url}
```

- 建需求时 Jira 已配 → 推 issue，回填 `external_key/external_url`。
- 未配 Jira → 手填 / 关联已有 key，平台照常工作（离线 / CI 无阻）。
- **不新增依赖**（复用已有 `httpx`）。

## API（`src/eddplatform/api/app.py`）

```
GET  /api/systems/{id}/requirements                列表（支持 ?key= 按 Jira 号过滤）
GET  /api/requirements/{req_id}                    详情
POST /api/systems/{id}/requirements                新建（Jira 可用则推送回填 key）
GET  /api/systems/{id}/dataset?requirement=R-101   用例按需求过滤（现有端点加 query）
GET  /api/comparison                               返回体新增 by_requirement 汇总
```

新建端点在脚手架层面追加到内存示例列表（与现有只读示例数据风格一致）。

## 不在本版范围（未来）

- 从 Jira **拉取**状态 / 字段（本版只写）。
- 平台内「按状态筛未完成需求」（本版去 Jira 筛）。
- 需求 ↔ Evaluation 的显式绑定（当前可由 Case 反推需求通过率，暂不显式建模）。
- 全生命周期编排（需求→设计→开发→评估→发布 的阶段/状态机）。

## 验收标准

1. 领域模型含 `Requirement` + 两处 `requirement_ids`；`pytest` 全绿。
2. `rollup_by_requirement` 正确算出达标 / 回归，且只统计两版共有用例。
3. Jira 未配置时全流程（建需求手填 key、跑评估、看需求汇总）不受阻。
4. API 能列需求、按 Jira 号过滤、返回体带 `by_requirement`。
5. 前端与原型一致：需求页、用例库需求列/过滤、对比页「按需求」汇总。
