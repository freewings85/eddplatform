## 组件 A · Temporal 编排 → live

把 `run_release_evaluation` 的编排逻辑搬到真实 Temporal workflow + activities，对 docker Temporal server 端到端真跑，逻辑与纯 Python 版一致，只换执行引擎（可重试 / 可观测 / 可断点续跑）。

### A.1 序列化与依赖注入（关键设计点）

Temporal workflow 的输入/输出、activity 的入参/出参必须**可序列化**。

- **可序列化的**：`SystemVersion` / `Module` / `Dataset` / `Case` / `Comparison` / `EvalResult` / `Requirement` 均为 pydantic v2 → 用 temporalio 的 `pydantic_data_converter`（`temporalio.contrib.pydantic`）配置 Client，直接传对象。
- **不可序列化的运行时依赖**：`provider`（`EnvironmentProvider`）、`target_factory`（`Callable`）、`evaluators`（`dict[str, Evaluator]`）。**解法：用类实例的 activity 方法**，把这些依赖闭包进 activity 实例——它们不进 workflow 输入，只在 worker 侧绑定。

```python
class ReleaseEvalActivities:
    def __init__(self, provider, target_factory, evaluators, modules):
        ...
    @activity.defn
    async def create_env(self, version: SystemVersion, otel_endpoint: str, ttl_hours: float) -> str:
        manifest = render_manifest(self.modules, version, otel_endpoint=otel_endpoint)  # 内部渲染
        return await asyncio.to_thread(self.provider.create, manifest, ttl_hours)
    @activity.defn
    async def evaluate_version(self, version: SystemVersion, cases: list[Case], env_id: str) -> EvalResult: ...
    @activity.defn
    async def destroy_env(self, env_id: str) -> None: ...
```

`create_env` 在 activity 内调用 `render_manifest(self.modules, version, otel_endpoint=...)` —— `modules` 由 worker 侧绑定、不进 workflow 输入；`otel_endpoint` 显式传入（消除 `render_manifest` 读环境变量的非确定性），故 workflow 无需持有 `modules`。`provider` / `target_factory` 可能是同步阻塞（kubectl、HTTP、LLM）→ 在 activity 里用 `asyncio.to_thread` 包，避免堵 event loop。

### A.2 Workflow

```python
@workflow.defn
class ReleaseEvaluationWorkflow:
    @workflow.run
    async def run(self, req: ReleaseEvalRequest) -> ReleaseEvaluationResult:
        baseline = await self._eval_one(req.baseline_version, req.dataset, req.cleanup)
        candidate = await self._eval_one(req.candidate_version, req.dataset, req.cleanup)
        cmp = compare(baseline, candidate)            # 纯确定性，留在 workflow
        if req.requirements:
            cmp.by_requirement = rollup_by_requirement(...)
        return ReleaseEvaluationResult(comparison=cmp, baseline=baseline, candidate=candidate)
```

- `ReleaseEvalRequest`：可序列化输入 = `{baseline_version, candidate_version, dataset, requirements, cleanup}`（`modules` 由 worker 侧绑定，因为 activity 要用它 `render_manifest`；`dataset` 走输入以便同一 worker 跑不同数据集）。
- `_eval_one`：`render_manifest`（纯函数，可在 workflow 内直接算，无副作用/无 env 读——**注意**：`render_manifest` 读 `EDD_OTEL_ENDPOINT` 环境变量，非确定性 → 改为在 activity 内渲染，或把 endpoint 作为 `ReleaseEvalRequest` 字段显式传入。**采用后者**：endpoint 进请求，`render_manifest` 在 activity `create_env` 内调用）→ `create_env` activity → `evaluate_version` activity → `destroy_env` activity（放 `finally`/`workflow` 补偿，保证 ephemeral：无论评估成败都销毁）。
- 纯确定性步（`compare` / `rollup_by_requirement`）留在 workflow，与 `pipeline.py` 结果逐字节一致。

### A.3 Worker 与 runner

- `build_worker(*, client, task_queue, provider, target_factory, evaluators, modules) -> Worker`：构造 activity 实例 + 注册 workflow，替换现有抛错的壳。
- `run_release_evaluation_via_temporal(*, client, task_queue, request, provider, target_factory, evaluators, modules) -> ReleaseEvaluationResult`：便捷入口——起（或复用）worker、执行 workflow、返回结果。签名尽量对齐 `pipeline.run_release_evaluation`，让二者可互换。
- `available()` 维持（反映 temporalio 是否装）。

### A.4 部署

`deploy/temporal/docker-compose.yml`：
- `temporalio/auto-setup`（含 server）+ `postgresql`（持久化）+ `temporalio/ui`（:8233 或 :8080 可视化）。
- 走 docker.io mirror（见记忆 k3d-local-cluster-howto，宿主已配 `docker.1ms.run`）。
- `deploy/temporal/README.md`：一键 `docker compose up -d` + 环境变量（`TEMPORAL_ADDRESS=localhost:7233`）+ 跑 demo 步骤。

选型：`auto-setup` docker 栈（与 Langfuse 部署一致、带 UI）优于 `temporal server start-dev` CLI（更轻但要装二进制、无持久 UI）。

### A.5 验证

- **单测（不依赖 server，进 CI）**：temporalio 自带 `WorkflowEnvironment.start_time_skipping()`（或 `start_local`）跑真 workflow + activities（用 `MockProvider` + 进程内 target + 本地兜底评分器），断言与 `test_orchestration.py::test_release_evaluation_end_to_end` 同样的结果（改善 1 / 回归 0 / applicable 3 / R-101 汇总）+ ephemeral 销毁 + `cleanup=False` 保留。**证明 Temporal 版与纯 Python 版语义等价。**

## 组件 B · Harbor 客户端补全（壳→完整，live 待实例）

在 `integrations/harbor.py` 现有 `available` / `image_ref` / `tag_exists` 上补：
- `list_tags(project, repo) -> list[str]`：列 artifact tags。
- `artifact_digest(project, repo, tag) -> str`：取 digest（按 digest 钉死，可复现部署）。
- `verify_pins(modules, version) -> None`：对 `SystemVersion.module_pins` 逐个校验 tag 存在；缺失时抛列出所有缺失 `<module>:<tag>` 的可操作错误。**未配置 Harbor（`available()` False）时直接返回/跳过**，不阻断离线流程。可选挂进 pipeline 的预检位（默认关，配了 Harbor 才生效）。

约束：复用 `httpx`，零新依赖。所有函数未配置时给可操作错误或安全跳过。

**测试**：`httpx.MockTransport` 造 Harbor v2 API 响应，覆盖 tag 存在/不存在、`verify_pins` 全过/部分缺失、未配置跳过。

## 组件 C · Backstage 客户端补全（壳→完整，live 待实例）

在 `integrations/backstage.py` 现有 `available` / `register_component` 上补：
- `component_descriptor(module) -> dict`：由平台 `Module` 生成 Backstage `kind: Component` 实体（`metadata.name`、`spec.type=service`、`spec.lifecycle`、`annotations`（git url））。

**测试**：`httpx.MockTransport` mock catalog API，覆盖 descriptor 生成正确、注册成功、未配置报错。
