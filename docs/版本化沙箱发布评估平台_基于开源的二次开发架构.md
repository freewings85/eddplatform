# 版本化沙箱 + 发布评估平台：基于优秀开源的二次开发架构

> 版本 v1.0 ｜ 日期 2026-07-02 ｜ 负责人：AI 团队开发负责人
> 证据来源：第 6 轮 deep research（对抗式交叉验证，全部一手/CNCF 来源）。
> 原则：**复用优秀成熟开源 + 走标准协议 + 二次开发尽量薄，不从头造轮子。**
> 关联：`开发迭代管理系统_开源选型清单与集成.md`、`AI业务全流程研发体系_SOP.md`。

---

## 核心结论

1. **地基就是 Kubernetes。** 你的直觉对——验证下来推荐的每一层要么 k8s 原生、要么跑在 k8s 上（Garden/vCluster/Okteto/Harbor/Backstage）。k8s 对你们**不重反而正合适**：你熟，且 `plan.txt` 已规划 K8s + Istio + 高危 Tool 沙箱，这平台正好长在那底座上。
2. **六层全部用成熟开源**，几乎不用从头开发。
3. **唯一要"二次开发"的只有三块薄胶水**：① Temporal 流水线代码 ② Backstage 自定义 Scaffolder action ③ 各服务 OTel 埋点 + Langfuse 数据集/评分配置 +「系统版本=镜像 tag 清单」manifest。
4. **微 VM 级黑盒隔离多半用不上**——namespace+NetworkPolicy(+vCluster) 通常够；Kata/gVisor 只留给"高危工具执行"（对应你 plan.txt 的高危 Tool 沙箱）。

## 分层架构（全部优秀开源，标注复用/自研）

| 层 | 选型（优秀开源）| 角色 | 成熟度/License | k8s 关系 | 复用/自研 |
|---|---|---|---|---|---|
| **① 一次性多服务环境** | **Garden**（+ **vCluster** 或 **Okteto** 加强隔离）| 按版本拉起整系统 env、跑完销毁；action-graph 管服务依赖、Test action 跑端到端 | 高，OSS，CLI 驱动 | **k8s 原生** | 直接用 |
| **② 编排控制器** | **Temporal** | 驱动 选版本→建env→跑用例→评估→对比→销毁；durable、可重试/重放/暂停 | 高，一手；Netflix Spinnaker 用它 | 跑在 k8s | **自研流水线代码（薄）** |
| **③ 黑盒隔离** | namespace+**NetworkPolicy** → **vCluster** → **Kata/gVisor** | 分级隔离 | 高（微VM层中等）| **k8s 原生** | 直接用（配置）|
| **④ 版本/镜像仓库** | **Harbor** | 「系统版本=一组镜像/Helm-chart tag」；签名/扫描；REST+Swagger API | **CNCF 毕业**，Apache-2.0，~30.5k★ | k8s 生态 | 直接用 |
| **⑤ 统一门户（一个入口）** | **Backstage** | catalog + Scaffolder 多步模板 + 插件 embed；SSO(内置多 provider + 通用 OAuth2/SAML) | 高，CNCF | k8s 生态 | **自研 Scaffolder action（薄）** |
| **⑥ 系统级 trace + 评估对比** | **OpenTelemetry → Langfuse** | 跨服务 trace 直发 Langfuse OTLP 端点；dataset run 自动生成、UI 并排对比 v1/v2 | 高，一手 | 无关 | **自研埋点+数据集配置（薄）** |

> 说明：Argo CD/Flux（GitOps 部署）、Kurtosis、Dagger、Argo Workflows、Jaeger/Tempo 本轮**未验证**（不代表不好，见开放问题）。

## 端到端流程（v1/v2 一次性沙箱，跑在 k8s 上）

```
Backstage 门户（唯一入口）
   │  发起「发布评估」：选版本组合(v1=原5服务 / v2=2新+3旧)
   ▼  (自定义 Scaffolder action 调用)
Temporal 工作流（durable，可重试/重放）   ← ① 自研薄胶水
   1. 从 Harbor 取镜像 tag → 渲染「系统版本」manifest(Helm/Kustomize)   ← ③ 自研manifest
   2. Garden 拉起隔离环境：namespace release-v1 / release-v2 (+NetworkPolicy 黑盒)
        └ 需更强隔离 → vCluster（每版本一个虚拟 k8s）
        └ 高危工具执行 → Kata/gVisor RuntimeClass
   3. 等就绪 → 跑用例套件，端到端打系统入口（Garden Test action）
   4. 各服务 OTel 埋点 + 上下文传播 → 全链路 trace 直发 Langfuse   ← ② 自研埋点
   5. Langfuse 评分 → dataset run(v1) vs run(v2) 并排对比（improved/regressed）
   6. 销毁环境：kubectl delete namespace / vCluster（TTL，ephemeral）
```

- **Istio 加成**（复用你 plan.txt 已规划的）：v1/v2 之间可做流量切分、故障注入，让发布评估更贴近真实。

## 唯一要"二次开发"的三块薄胶水（其余全是配置/复用）

1. **Temporal 流水线代码**：把「选版本→建env→跑用例→评估→对比→销毁」写成 workflow + activities。引擎不自研，只写这条流程；注意 workflow 需确定性（时间/随机/外部调用放 activity 里）。
2. **Backstage 自定义 Scaffolder action**：Backstage **没有**开箱的 build-env/test/destroy action，需写一个自定义 action（TypeScript handler）去调 Temporal/Garden/Harbor/Langfuse 的 API。门户平台不自研，只写这个 action + 模板。
3. **各服务 OTel 埋点 + Langfuse 数据集/评分配置 +「系统版本=镜像 tag 清单」manifest**：跨 5 个服务（含非 LLM 服务）埋点并传播 context，才能把整系统一次运行拼成一条可评估的 trace；用例集/评估器在 Langfuse 里配。

> **判据**：这三块都是"配置 + 调 API 的胶水"，不是造平台。评估平台、编排引擎、环境编排、镜像仓库、门户、trace 后端——**一个都不自研**。

## k8s 隔离分级（按黑盒强度选）

| 级别 | 手段（k8s 生态）| 何时用 |
|---|---|---|
| 逻辑隔离（默认）| namespace/版本 + **NetworkPolicy** + RBAC + ResourceQuota | 默认；最轻，多数够用 |
| 控制面隔离 | **vCluster**（每版本一个虚拟 k8s，独立 API endpoint）| v1/v2 要像独立集群、互不干扰 |
| 运行时微 VM 黑盒 | **Kata Containers**（可选 Firecracker 作 VMM）/ **gVisor** | 跑不可信/高危工具（= plan.txt 高危 Tool 沙箱）|

> ⚠️ **Firecracker 不能直接进 k8s**（不是 CRI 运行时），要通过 **Kata** 用它当 VMM。多数内部发布评估用 namespace+NetworkPolicy(+vCluster) 即可，**微 VM 别过度设计**。（网上那些 gVisor/Kata/Firecracker 的启动时间/安全排名数字本轮被证伪，勿引用。）

## 与你们 plan.txt 的契合
- **K8s + 容器化 + Istio**：本架构直接长在其上；Istio 复用于 v1/v2 流量切分与故障注入。
- **高危 Tool 沙箱（权限/网络/文件隔离/执行审计）**：对应第 ③ 层 Kata/gVisor + NetworkPolicy。
- **灰度发布闭环**：发布评估通过 → 走你们 APISIX/Istio 灰度（+ 前述 OpenFeature/Unleash flag）。

## 落地 MVP → 路线
1. **MVP（单版本跑通）**：k8s 上用 Garden 定义 5 服务 + 一个 namespace 环境；Harbor 存镜像；手动跑一次用例套件 → OTel → Langfuse 看 trace/评分。
2. **加对比**：同一用例集跑 v1、v2 两个环境 → Langfuse dataset run 并排对比。
3. **加编排**：Temporal 写「建env→跑→评→销」workflow，一键跑完自动销毁。
4. **加门户**：Backstage 自定义 Scaffolder action，唯一入口发起 + 看对比。
5. **加隔离/灰度**：vCluster 强隔离、Kata 跑高危工具、Istio/flag 灰度。

## 证据强度与开放问题（诚实）
- **高置信（多条 3-0 一手/CNCF）**：Garden、Temporal、Harbor、Backstage、OTel→Langfuse。
- **中等**：vCluster/Okteto（3-0，但一条主引用是竞品博客+官方佐证）；微 VM 隔离（2-1，学生论文+AWS/Kata 佐证；对比数字已证伪勿用）。
- **本轮未验证（需再定）**：
  - **GitOps 部署/回滚 Argo CD vs Flux** 放在哪层（大概率作 Temporal 的一个 activity 或替代部署步）——待定。
  - **Temporal vs 更轻的 Argo Workflows/Dagger**：你们熟 Temporal 可直接用，但要掂量运维一个 Temporal 集群的成本。
  - **跨 5 服务（含非 LLM）拼成一条可评估 trace**：靠 OTel 上下文传播，系统级规模未被独立验证，是实打实的集成活。
  - 微 VM 隔离到底要不要：内部评估多半 namespace+NetworkPolicy(+vCluster) 就够。
- 时效：Backstage 前端在迁移（自定义步骤布局暂不支持新前端）；Langfuse 自托管 OTLP 有过 hang bug(#9900)；Harbor 原生 ChartMuseum 已弃用改 OCI；Garden 一键 preview 是 Cloud(非 OSS)、CLI 路径才是 OSS。
