01 · 根子

今天的形态是「以系统为中心的发布评估台」

进平台先选系统，系统底下挂着模块、版本、需求、用例库、评估器、环境，然后才是运行 / 评估 / 对比。 需求在里面只是「资产」里的一项，和用例库、评估器平级——它是个附属品。

可一个需求「做对没有」，恰恰是这个平台唯一要回答的问题。所以需求得从资产里提出来，成为系统底下的主线； 用例是它的验收标准，评估是它的证据，门禁是它的结论。系统退成需求的运行底座，但第一级导航仍然是系统。

02 · 骨架

goalplatform 是独立产品，不打通数据、不合并代码——这里只借它已经验证过的形态。 它的骨架恰好就是 EDD 需要的：一级是组织维度，二级是一棵树 + 一个对齐的右视图。

05 · 生命周期

四段落在哪：三段已有，一段真空

阶段在 EDD 里是什么现状
需求 需求树 + 挂上验收用例。取代现在扁平的 Requirement 薄锚点，参考 Goal 改成递归树 + 负责人 + 一对一 Jira issue。 有雏形 · 要改成树
测试 单独运行——自测、不评分。RunType 里本来就把「单独运行」和「评估产生的运行」分开了，这正好就是测试与评估的分界。 已有
评估 正式跑分 + 老新对比（改善 / 回归 / 持平，只统计两版共有用例）+ 需求达标卷积。Langfuse 存分与真实 token。 已有 · 最强
发布上线 门禁：无回归 + 需求达标 → 放行。今天的流水线评完就把环境销毁，从没往生产推过一次，也没有「放行 / 拦下」这个动作。 真空
06 · 借用边界

从 goalplatform 拿什么，不拿什么

拿它的设计与方案

  • ✓
    账号：Jira 登录那套。用户名密码 → 调 Jira 校验 → 自动建用户；密码 Fernet 加密存、任何接口不回显；会话走签名 cookie。EDD 现在是 0。
  • ✓
    存储：SQLModel + MySQL。本地不配环境变量即回退 SQLite；表启动自动建。EDD 现在的 STORE 是纯内存，重启就没。
  • ✓
    需求模型：参考 Goal 的形态。递归父子树、有负责人、一对一挂 Jira issue。数据是 EDD 自己的，不和 goalplatform 打通。

明确不拿

  • ✕
    固定五阶段流水线。业务需求确定 / 方案确定 / 开发完成 / 测试 / 发布上线——那是进度视角。
  • ✕
    甘特、周期、起止日期。EDD 重心不在进度时间上，一个日期字段都不碰。
  • ✕
    人工阶段状态与审批。EDD 的结论由证据算出来，不由人点「已完成」。
  • ✕
    Goal 数据本身。goalplatform 是独立产品，两边不同步、不共库。
待你定

「发布上线」这段，EDD 做到哪一步？

四段里只有这段是真空的，它决定产品形态的终点——也决定上面那条「🚦 发布门禁」到底是个结论页，还是一个能按下去的按钮。

  • A · 只出门禁结论，不亲自 ship给出「可放行 / 拦下」+ 证据链接，真正推生产由现有 Jenkins 干。符合仓库「不造轮子」口径，不碰生产发布权，风险最小。
  • B · 门禁 + 亲自推到生产评过了直接推上生产、记发布记录、支持回滚。链路真闭环；代价是 EDD 拿到生产发布权，要做权限 / 审计 / 回滚。
  • C · 连门禁都不做，只出报告停在评估对比报告，放不放行人自己看。改动最小；但「发布上线」就不算 EDD 的一段，全生命周期实际是三段。
