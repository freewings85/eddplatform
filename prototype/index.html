账号方案（参考 goalplatform 的做法）

登录方式填 Jira 用户名 + 密码 → 后端拿去 Jira 校验（/rest/api/2/myself）→ 通过即登录，用户由登录自动创建

为什么要存密码之后要代表该用户调 Jira（建 issue / 关联需求）

存储方案（参考 goalplatform 的做法）

生产MySQL（公司已有实例）· 设了 *_MYSQL_HOST 即启用 · 库要预先建好，表由应用启动自动创建
本地开发不配任何变量即回退 SQLite（单写库，只能单实例）
失败姿态MySQL 连不上启动失败（fail fast），不静默回退 SQLite——避免生产数据悄悄写进本地文件
容器无状态、可多副本；加密密钥存库，不挂卷

系统版本

系统版本 = 各模块钉住一个 tag/commit 的静态快照（拉自 Harbor / Git）。造版本只做一件事：钉 tag。

🧩 系统版本就是版本，别的都不掺：一个版本 = 各模块钉住 tag 的静态快照，它自己不会跑，与用例集 / 任务 / 确认启动无关。 「新建版本」只选各模块的 tag；跑评估是之后、且在别处的事——在下方每行的「▶ 发起评估」上发起（拿一个评估任务定义去跑这个版本），不塞进新建里。

需求树 · 保险报价系统

需求是"为什么做这个版本"的锚点，可递归成树。详情（描述 / 状态 / 流转）在 Jira（唯一真相源），平台只存锚点 + 挂接。

🏷️ 需求与用例是「标签」关系，不是从属关系：一条用例可打多个需求标签，一个需求也被多条用例覆盖（多对多）。 用例库、运行、评估都和需求平级——标签只用来按需求过滤用例、以及把用例结果卷成需求达标。
🌳 树上的状态点是达标，不是进度——绿=达标 / 红=有回归 / 黄=证据不足 / 灰=还没打过标签。达标沿树往上卷：子需求全达标，父需求才算达标。「做到哪一步了」不归这里管。

发布记录

一次发布 = 选一份评估报告（发布依据）→ 把该报告评的那个系统版本整套打成包 → 产出可部署的压缩包。发布记录留痕：谁、依据哪份报告、发了哪个包。

📦 发布包 = 整套系统，不是单个模块：一个系统版本钉住了全部 5 个模块的 tag，打出来的包必须原样带上这 5 个模块 + 基础服务。 形态用 Helm chart（helm package 产出 .tgz 压缩包）——SystemVersion.module_pins 天然就是 chart 的 values.yaml， 现有 render_manifest 渲染的 manifest 天然就是 templates/。包推到 Harbor（OCI chart 仓库，客户端已就绪）。

发布记录是追溯的落点：从一条发布 → 依据的评估报告 → 该报告的运行记录 / 轨迹 → 用例 → 需求标签，整条链路可回溯。包里钉的是 digest 不是浮动 tag，同一个包在任何环境部署结果一致。

用例集 · 保险报价 (200)

用例有自身版本（编辑历史）+ 适用系统版本（多版本通用 / 某版本专属）

Dataset用例集pydantic-evals 概念

框架原义：「为评估某个 Task 而设计的一组 Case 的集合」——它是评估的题库，一个 Dataset 可以被反复用于多次 Experiment。

在 EddPlatform：一个系统一个用例集。老新版本必须跑同一个 Dataset，对比才成立。 我们加的：每条 Case 带「适用系统版本」——框架不需要（它只评一个 Task），我们要跨版本比，就必须能排除"某版本专属"的题，否则对比不公平。

Case用例pydantic-evals 概念

框架原义：「一个测试场景，对应 Task 的一组输入，可选带期望输出、metadata、以及该用例专属的 evaluators」。

在 EddPlatform：下表每一行就是一个 Case。字段一一对上——inputs / expected_output / metadata / 专属评估器都有。 我们加的：case_version（用例自身的编辑历史）、applicable_versions（适用哪些系统版本）、requirement_ids（需求标签）。

⚖️ 评估对比时，只统计"被比两个版本都适用"的用例；某版本专属用例（如 v2 才有的功能）会单列、不计入对比，避免不公平。
🏷️ 需求是用例上的标签（多对多）：一条用例可打多个需求标签。标签只用来过滤（下面这排）和把结果卷成需求达标——用例库和需求树是平级的两份资产，不是谁属于谁。

评估任务定义 Task

"怎么评这个系统" 的可复用定义：绑入口 + 用例集 + 评估器 + 环境。版本无关——运行时才选版本。

Task评估任务定义 · 被评对象pydantic-evals 概念

框架原义：「被评估的函数或系统：接收 Case 的 inputs，产出 output」。框架里 Task 就是一个函数， 传给 dataset.evaluate_sync(task) 就跑。

我们和框架的分道点，就在这一个对象上。框架是 code-first——版本、入口、依赖全闭包进那个函数里，改一版就改代码。 我们要跨版本比、且可在 UI 管理，所以把 Task 从"一个函数"拆成一份声明式定义：绑系统 + 入口适配器（怎么喂输入 / 读输出）+ 用例集 + 评估器 + 环境， 但刻意不含版本。系统版本是运行时参数：同一个 Task 定义在 v1、v2 各跑一次，就得到两个 Experiment，绑成一个 Comparison。

评估器（给用例打分 / 判定的规则）

用例跑完有个输出，"评估器"就是判断这个输出算不算通过的规则；引擎复用 Langfuse（框架无关，不自研），下面的代码写法是可选后端(pydantic-ai 项目)

Evaluator评估器pydantic-evals 概念

框架原义：「分析并给 Task 在某个 Case 上的结果打分」。可以是确定性的（代码校验）或概率性的（LLM 判断）。 返回值自动归类：bool → assertion（通过/不通过）、数值 → score、字符串 → label——这条规则我们逐字对齐。

作用范围也对齐：Dataset 级评估器跑全部 Case，Case 级评估器只跑挂它的那条。

在 EddPlatform：评估器是"尺子"，老新版本必须用同一把，对比才公平——所以我们 加了 version（框架的 get_evaluator_version()），并在评估对比前校验两侧尺子一致。 框架是 code-first（评估器写在 git 里），我们要可在 UI 管理——这是我们和它必然分道的一处。

🔁 打分流程：用例输入 → 系统跑出 输出 + 轨迹(ctx) → 评估器 evaluate(ctx) → 返回值自动归类：bool→通过/不通过、数值→分数、字符串→标签。一条用例可挂多个评估器，全过才算过。
code 代码评估器

写死的校验，结果确定可复现。用内置（EqualsExpected / Contains / MaxDuration / HasMatchingSpan…）或自定义 Evaluator 子类。

LLM-judge 模型评委 LLMJudge / GEval

让模型按 rubric 打分，处理主观问题（幻觉/语气）。默认出"通过+理由"，可再附 0-1 分。⚠️ 需人工校准。
GEval：给 criteria + 显式 evaluation_steps，评委先出推理轨迹再打整数分（G-Eval，Liu et al. 2023）——把"怎么判"写死成步骤，比裸 rubric 稳。

🧭 先找内置的，别自己写——尤其轨迹类。我们评的是 agent，"工具调没调对"是核心维度，而这类全是内置的， 且比手写的强：TrajectoryMatch 用 LCS 算 F1 给部分分，"少调一个工具"和"完全跑偏"不同分；手写的 bool 判定两者一样惨。

🔒 评估器是尺子，尺子必须钉住版本：平台的核心主张是"老新版本用同一套评估器，对比才公平"。 可 rubric / 代码是能被人改的——改完之后，v1 的分和 v2 的分就不可比了。所以每个评估器带 version， 改判定逻辑必须 bump；评估对比时校验两侧尺子一致，不一致就拒绝出「改善 / 回归」结论。 （用例有 case_version、版本有 pin，尺子却没版本——这是个漏洞。）

框架怎么做

写 Python 类继承 Evaluator、实现 evaluate(ctx)。 代码在你的 git 里——可 review、可 diff、有 commit。code-first。

我们该怎么做 引用一个「版本化 evals 包」，不执行字符串

代码 + 它的依赖是一个钉死版本的构建产物（pip 包 / 镜像），平台只存引用：
source: "myproj-evals@1.4.0" · ref: "注入检测" · params: {patterns:[...]} · version: "v2"
UI 管"用哪个包的哪个类 / 什么参数 / 什么版本"，从不粘代码。

📦 为什么不能只存 ref+params：代码 import 的包，得在评估器运行的环境里装好。 实测一个 import sentence_transformers 的自定义评估器——类能实例化、{threshold:0.9} 能序列化， 但 evaluate() 一跑就 ModuleNotFoundError。所以自定义评估器不是代码片段，是一个带 pyproject 依赖的版本化包。

版本钉的是两样：评估器逻辑 + 它那套依赖（像 lockfile）——正如钉一个模块 tag = 钉住代码+依赖。 在哪跑分两档（同沙箱思路）：可信 → evals 包 pip 进 worker；要隔离（依赖冲突/不可信）→ 依赖烤进镜像、评估器在自己容器里跑。 ⚠️ Langfuse 沙箱代码评估器有包限制 + 2s 超时——sentence-transformers 这类重依赖跑不进它的沙箱，得走我们自己的 worker。 代价：自定义评估器不能在 UI 里现写，得提交代码 + 声明依赖。这代价是对的——一把影响发布决策的尺子，本就该走代码评审。

能否与 pydantic-evals 一一对应？　—— 不能，而卡点不是"少几个"

🧩 它 13 个内置，我们兜底引擎 5 个。但数量不是问题——engine.py 只是"零依赖本地兜底"（离线 / CI 冒烟），生产走 Langfuse，兜底不必复刻 13 个。
真问题：EvaluatorDef 存不下它们的参数。我们只有 5 个字符串字段 + 1 个 float，而它们要 list[str] / dict / 多个 bool 开关。
评估器它需要的参数我们的 EvaluatorDef 往哪放

💡 答案在它自己身上：它序列化评估器用 EvaluatorSpec，字段只有两个——name: str + arguments: dict|tuple|None， 就是 YAML 里那个 - TrajectoryMatch: {expected_trajectory: [...], order: in_order}。名字 + 任意参数 dict，13 个全能表达。
所以修法：EvaluatorDef 加 params: dict，与 EvaluatorSpec 同构，rule/threshold/rubric 退成 params 里的 key。 这样加新评估器不用改模型——和 metrics 用开放 dict 是同一个道理：能扩展的地方留 dict，别写死成字段。

🧭 Langfuse 把"评估"分两层：
① 评估器 → Score（你定义标准的地方）：LLM-as-a-Judge（主观质量）/ Code evaluator（确定性校验）/ SDK 自定义 Score。

① 评估器 → Score

LLM-as-a-Judge：写 rubric 让模型打分（主观：质量/幻觉/语气）。Code evaluator：Python 函数确定性校验（exact match / JSON schema / 工具调用 / 自定义规则，2s 内跑完）。均可跑在 experiment / trace / 单个 observation 上。

② 可观测指标

延迟 / 成本 / token / TTFT 由 Langfuse 自动采集（TTFT 需 emit completion_start_time）。看聚合走 Metrics API / Latency & Cost Dashboard；要做成发布门禁 → SDK create_score 回灌一条 Boolean。

Experiment评估任务运行实例（一次实验）pydantic-evals 概念

框架原义：把一个 Task 跑遍 Dataset 的全部 Case 的一次执行（dataset.evaluate_sync(task)）， 产出一份 Report。关系：Experiment ── (1) Task、Experiment ── (Many) Evaluator、 Experiment ── (Many) Case results、Dataset ── (Many) Experiment。

Data Flow 五步：定义 Dataset → 执行 Experiment → 逐 Case 跑 → Evaluator 打分 → 汇成 Report。下表每一行就是一次 Experiment。

在 EddPlatform：Experiment = 「一个评估任务定义（Task）」运行在「一个系统版本」上的一次实例。 Experiment ── (1) Task 依然成立——只是我们的 Task 是声明式定义，且版本在运行时才注入。 所以一次运行要多做框架没有的两件事：建/销一次性环境、把结果登记成可被 Comparison 引用的对象。 同一个 Task 定义跑 v1、v2 两次 → 两个 Experiment → 一个对比。

Case results逐用例结果pydantic-evals 概念

框架原义：一次 Experiment 对每条 Case 产出一个结果（Experiment ── (Many) Case results）—— 记录该 Case 的实际 output、各评估器打的分、通过与否、耗时。Report 就是把它们汇总成一行行 + 一个通过率。

在 EddPlatform：模型里就是 CaseResult{case_id, passed, assertions, scores, labels}（EvalResult.case_results 列表）。 下表是 #E-3002 的逐 Case 结果——它是对比、需求达标、发布依据的最小事实颗粒，往上每一层都由它卷出来。

Comparison评估对比框架没有 · 我们加的

pydantic-evals 的数据模型里没有这个对象——它的 Data Flow 走到「汇成一份 Report」就结束了， 一个 Experiment 只评一个 Task，天然不存在"两版比"的问题。

但它留了接口：Dataset ── (Many) Experiment（同一题库可被反复跑）。 Comparison = 把其中两次 Experiment（v1 的和 v2 的）绑成一个对象，再往上卷成需求达标、变成发布依据。这是 EddPlatform 的立身之处。

公平性靠三条铁律：① 只统计两版都适用的 Case；② 两侧用同一把尺子（评估器版本一致，见下方前置校验）；③ 两版跑各自独立的一次性沙箱。 任一条破了，「改善 / 回归」就是假结论。
