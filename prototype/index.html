评估器（给用例打分 / 判定的规则）

用例跑完有个输出，"评估器"就是判断这个输出算不算通过的规则；引擎复用 Langfuse（框架无关，不自研），下面的代码写法是可选后端(pydantic-ai 项目)

🔁 打分流程：用例输入 → 系统跑出 输出 + 轨迹(ctx) → 评估器 evaluate(ctx) → 返回值自动归类：bool→通过/不通过、数值→分数、字符串→标签。一条用例可挂多个评估器，全过才算过。
code 代码评估器

写死的校验，结果确定可复现。用内置（EqualsExpected / Contains / MaxDuration / HasMatchingSpan…）或自定义 Evaluator 子类。

LLM-judge 模型评委 LLMJudge

让模型按 rubric 打分，处理主观问题（幻觉/语气）。默认出"通过+理由"，可再附 0-1 分。⚠️ 需人工校准。

可选后端(pydantic-ai) 内置评估器： EqualsExpectedEqualsContainsIsInstanceMaxDurationHasMatchingSpanLLMJudge （Python 评估器已被移除，不提供）

本系统的评估器

评估器定义方式读取 (ctx)输出通过阈值用在用例
金额校验自定义 codeoutput.premiumassertion== 规则引擎#17 #88 #91
from dataclasses import dataclass from pydantic_evals.evaluators import Evaluator, EvaluatorContext @dataclass class 金额校验(Evaluator): def evaluate(self, ctx: EvaluatorContext) -> bool: return ctx.output["premium"] == ctx.expected_output["premium"] # 返回 bool → 自动记为 assertion（通过 / 不通过）
确定性：同输入永远同结果，适合金额/字段/格式这类有标准答案的检查。也可用内置 EqualsExpected 免写代码。
条款解释LLM 评委output（+ inputs）assertion + scorejudge ≥ 4#63
from pydantic_evals.evaluators import LLMJudge LLMJudge( rubric="只依据真实条款，不得编造优惠/减免；1=严重编造，5=完全准确", model="openai:gpt-5.2", include_input=True, assertion={"include_reason": True}, # 通过 / 不通过 + 理由 score={"evaluation_name": "条款质量"}, # 另附 0-1 分 )
⚠️ 上线前用人工标注过的用例校准 rubric，让判分与人一致（对应 #63 判出的 4.5→2.1 幻觉）。
工具调用正确内置 codespan_treeassertion命中工具 span#17 #44
# YAML（EvaluatorSpec）——挂到用例集 / 单个用例 evaluators: - HasMatchingSpan: {query: {name_contains: "tool:quote-engine"}} # 轨迹里存在该工具调用的 span → 通过（读 ctx.span_tree，来自 OTel）
工具/延迟这类"看执行过程"的判定，靠 OTel 轨迹：HasMatchingSpan 查 span、MaxDuration 看耗时。
延迟阈值内置 codedurationassertion≤ 3s#91
